# Happy Star 星图皮肤设计

日期：2026-07-06

## 1. 背景

Happy Star 当前版本已经实现家庭积分、任务打卡、奖励兑换、家长审批、日历、产能总览、任务/奖励行内编辑和安全部署。现有视觉是“阳光、明亮、儿童友好”的日常皮肤。

本设计新增一套可切换的高级视觉皮肤：**Lucky Star Map / 幸运星图**。它从用户提供的深色知识图谱类界面中提炼视觉语言：深色星空、发光节点、连线网络、玻璃面板、星轨动画和命令条式信息区。但它不复刻任何第三方产品的品牌、布局、文案或具体构图。

目标不是把 Happy Star 变成严肃政务/AI 平台，而是把家庭积分系统改造成一个“孩子通过努力点亮星图、抵达奖励星球”的体验。

## 2. 目标

1. 提供一套与当前阳光版并存的星图皮肤，而不是直接替换现有皮肤。
2. 保留现有业务逻辑和数据模型，不改积分、兑换、审批、产能算法。
3. 在实现前先完成高仿真 HTML 原型，作为视觉契约。
4. 让孩子端更有探索感和成就感，让家长端仍然清晰可编辑。
5. 避免只做“暗色模式”：必须具备星图节点、能量核心、奖励星球、玻璃面板和发光动效。

## 3. 非目标

- 不做 3D/WebGL 第一版。
- 不做复杂拖拽星图编辑。
- 不引入 Material UI、Ant Design 等重型 UI 库。
- 不把主题选择写入后端或 JSON 数据文件；第一版只用浏览器本地偏好。
- 不复制参考图的品牌、Logo、法律/AI 平台文案、具体布局或节点结构。
- 不牺牲任务打卡、奖励编辑、产能查看的可读性。

## 4. 核心隐喻

| 现有概念 | 星图皮肤表达 |
|---|---|
| 积分 | 幸运星能量 |
| 当前余额 | 能量核心亮度 |
| 任务 | 可点亮的任务星点 |
| 完成任务 | 星点点亮 + 星轨流光 + `+N 星能` |
| 奖励 | 可抵达的奖励星球 |
| 预计到手 | 航线抵达时间 |
| 产能总览 | 导航仪 / 星图航线计算 |
| 日历 | 星轨时间轴 |
| 家长管理 | Mission Control 家长控制台 |

## 5. 主题策略

新增主题：`sunny` 与 `starmap`。

第一版主题偏好存在 `localStorage`：

```text
happy-star-theme = sunny | starmap
```

默认策略：

- 新用户默认仍是 `sunny`。
- 登录页提供一个小型主题切换按钮。
- 切换后当前浏览器记住选择。
- 不影响服务器端数据，也不影响其他设备。

实现要求：

- 根元素增加 `data-theme="sunny"` 或 `data-theme="starmap"`。
- 所有主题色通过 CSS variables 控制。
- 现有阳光版视觉不能被破坏；星图样式必须包在主题选择器下。

## 6. 视觉语言

### 6.1 颜色

星图皮肤主色：

| 用途 | 建议色 |
|---|---|
| 页面背景 | `#061112` / `#091a1c` 深墨青 |
| 玻璃面板 | `rgba(12, 31, 34, 0.72)` |
| 面板边线 | `rgba(175, 238, 225, 0.18)` |
| 主星能 | `#f8d995` 香槟金 |
| 任务连线 | `#9ee8f2` 青蓝 |
| 奖励星球 | `#b99cff` 紫 |
| 王颢霖 | `#ff88b1` 粉 |
| 王仲贤 | `#7db8ff` 蓝 |
| 成功/完成 | `#7ff0c3` 青绿 |
| 警告/不足 | `#ffb86b` 暖橙 |

注意：

- 黄色不做大面积背景，只作为能量高光。
- 深色背景必须带层次，不能纯黑压抑。
- 表单输入区域要保证足够对比度。

### 6.2 质感

- 背景：径向暗青光晕 + 少量星尘粒子。
- 卡片：半透明玻璃、细描边、轻阴影、轻微内发光。
- 节点：圆形/星球形，有外环、微弱脉冲。
- 连线：细线为主，选中或完成时出现流光。
- 文字：主文字浅青白，次级文字低透明度青灰。

### 6.3 动效

第一版必须实现：

- 星点空闲微脉冲。
- 完成任务时任务节点点亮。
- 完成任务后连线出现 600-900ms 流光。
- `+N ⭐` 上浮继续保留，但样式改为星能浮标。
- 奖励申请成功触发小范围星尘，不做全屏爆炸式干扰。

必须避免：

- 大量闪烁。
- 背景粒子太密。
- 动画导致低性能设备卡顿。
- 表单输入时动画抢注意力。

## 7. 高仿真原型要求

实现真实代码前，必须先新增：

```text
docs/prototypes/star-map-skin.html
```

这个文件是视觉契约，不接真实 API，用模拟数据。它必须包含：

1. 登录页星图版。
2. 孩子任务页星图版。
3. 孩子奖励页星图版。
4. 家长任务/奖励管理 + 产能总览星图版。
5. 至少一个可点击的任务完成演示。
6. 至少一个可点击的奖励进度/兑换演示。
7. 横屏桌面视图优先，移动端给出合理堆叠。

原型验收标准：

- 第一眼有“高级星图系统”的感觉。
- 任务和奖励语义清晰，不只是漂亮背景。
- 家长表单可读、可编辑，不被暗色风格吞掉。
- 原型截图可作为后续实现对照。

未通过原型验收前，不进入真实代码实现。

## 8. 页面设计

### 8.1 登录页

结构：

- 背景是一片深色星域。
- 中间显示 Happy Star 标题和“越努力，越幸运”。
- 三个身份入口是三颗入口星球：王颢霖、王仲贤、家长。
- PIN 输入以玻璃卡片弹出。

行为：

- 切换身份时入口星球有轻微放大和光环。
- 输错 PIN 显示柔和错误提示，不用刺眼红色大面积警告。
- 顶部或右上角提供主题切换。

### 8.2 孩子任务页

结构：

- 顶部：头像、姓名、幸运星能量、切换按钮。
- 中央：幸运星能量核心。
- 任务节点围绕核心分布，任务列表仍保留在下方或右侧。
- 当前可打卡任务有发光按钮。

任务状态：

| 状态 | 表现 |
|---|---|
| 未完成 | 暗色节点，弱光外环，按钮“打卡” |
| 可重复完成 | 节点显示 `已完成/上限` |
| 达到上限 | 节点变亮，右侧对勾，按钮消失 |
| 完成瞬间 | 节点点亮、连线流光、`+N ⭐` 上浮 |

约束：

- 任务行布局仍沿用现有结构，避免孩子不知道点哪里。
- 不用把所有任务强行塞进中心星图；小屏可以星图在上、列表在下。

### 8.3 孩子奖励页

结构：

- 奖励以奖励星球/星舰卡片展示。
- 每个奖励显示：名称、emoji、积分价、当前进度、预计到手。
- 可兑换奖励显示金色/紫色高光和“申请兑换”。

行为：

- 积分不足时显示“还差 N ⭐”。
- 申请兑换成功后出现星尘和“已发送给家长审批”。
- 不改变现有“申请时不扣分，审批通过才扣分”规则。

### 8.4 家长任务管理页

结构：

- 左侧：任务管理表单和任务列表。
- 右侧：产能总览星图导航面板。
- 横屏左右分栏，竖屏上下堆叠，保持当前响应式约定。

视觉：

- 表单是深色玻璃面板，但输入框必须高对比。
- 产能三档：基础、现实 80%、满分 100% 做成三个导航仪指标。
- 奖励 ETA 表可视化为航线抵达时间。

行为：

- 新增/编辑/删除任务后，产能面板即时刷新。
- 核心任务用星标或能量环标识。

### 8.5 家长奖励管理页

结构同任务管理页：

- 左侧：奖励管理表单和奖励列表。
- 右侧：产能总览。

重点：

- 编辑奖励积分时，右侧预计到手立即变化。
- 让家长调积分时能直观看到“这个奖励太容易/太难”。

### 8.6 日历页

第一版只做轻量星图化：

- 月历格子保留。
- 今日格子有星环边框。
- 得分 chip 改成发光星点标签。
- 兑换日用奖励星标。

不把日历改成复杂星图，避免可读性下降。

## 9. 组件设计

新增/改造组件建议：

| 组件 | 作用 |
|---|---|
| `ThemeToggle` | sunny/starmap 切换 |
| `StarMapBackdrop` | 背景星尘、径向光、轻粒子 |
| `EnergyCore` | 幸运星能量核心 |
| `TaskConstellation` | 任务节点与连线 |
| `RewardPlanetCard` | 奖励星球卡片 |
| `GlassPanel` | 星图皮肤玻璃面板容器 |
| `StarmapCapacityPanel` | 产能总览的星图视觉包装 |

原则：

- 尽量复用现有 `TaskRow`、`RewardRow`、`CapacityPanel` 的数据逻辑。
- 星图组件只负责表现，不承载业务规则。
- 后端 API 不因皮肤变更而改变。

## 10. 数据流

现有数据流不变：

- 登录：`POST /api/login`、`GET /api/me`
- 任务：`GET /api/tasks`、`POST /api/tasks/:id/complete`
- 奖励：`GET /api/rewards`、`POST /api/rewards/:id/redeem`
- 产能：`GET /api/admin/capacity`
- 管理：`GET/POST/PUT/DELETE /api/admin/tasks|rewards`

主题数据流：

```text
App 启动
  -> 读取 localStorage happy-star-theme
  -> 设置 document.documentElement.dataset.theme
  -> 用户切换主题
  -> 更新 localStorage 与 data-theme
```

## 11. 可访问性与可用性

- 所有发光文字对比度必须足够。
- 不能只用颜色表达完成/不可用状态，仍保留文本和图标。
- 动画不应阻止操作。
- 小屏幕上星图装饰可以弱化，核心操作列表优先。
- 家长编辑表单必须比装饰更重要。

## 12. 性能约束

- 第一版不用 Three.js/WebGL。
- 背景粒子优先 CSS/SVG，必要时用轻量 canvas。
- 页面空闲动画数量可控。
- 低性能设备上关闭大面积模糊和密集粒子。
- 构建包体增长应保持可控；避免引入大型动画库。

## 14. 验收标准

视觉：

- 原型和实现都能明显看出“星图皮肤”，而不是普通暗色主题。
- 星图皮肤有深度、发光节点、玻璃面板和星轨动效。
- 不复制参考图的品牌和具体界面。

功能：

- 任务打卡、每日上限、积分滚动不回归。
- 奖励申请、家长审批、余额校验不回归。
- 任务/奖励编辑和产能总览不回归。
- 主题切换不影响登录态和数据。

可用性：

- 孩子能明确看到哪里打卡。
- 家长能明确编辑任务和奖励。
- 暗色下所有文字、按钮、输入框可读。

工程：

- 不引入数据库。
- 不引入重型 UI 库。
- 不改后端业务模型。
- 不提交 `data/`。
